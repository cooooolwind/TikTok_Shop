# 电商场景 AIGC 带货视频生成系统 — 项目实施计划（修订版）

## Context

基于《电商场景AIGC带货视频生成系统 - AI全栈挑战赛课题》文档，梳理出完整的项目实施计划。该课题要求构建一个面向 TikTok Shop 商家的端到端 AIGC 带货视频生成系统，核心链路为：素材库建设 → 剧本生成 → 视频创作，技术栈基于 React + Node.js + TypeScript + 火山引擎 OpenAPI。

---

## 一、项目概述

- **项目定位**：面向电商商家的 AIGC 短视频自动生成平台，覆盖从素材管理到成片导出的完整链路
- **核心价值**：让商家通过输入商品链接/图片，端到端自动生成一条 15s 以内的带货视频
- **技术边界**：React 前端 + Node.js 后端 + TypeScript 全栈 + 火山引擎模型 API
- **团队规模**：2-3 人
- **开发周期**：4 周（含缓冲），严格按 MVP → 进阶 → 加分 分层交付

---

## 二、技术选型

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | React 18 + TypeScript + Vite | 课题要求，Vite 保证开发体验 |
| UI 组件库 | Ant Design 5 | 后台类页面开箱即用，减少 UI 开发量 |
| 状态管理 | Zustand | 轻量，适合中等复杂度 |
| 图表库 | ECharts | 数据看板可视化 |
| 后端框架 | NestJS + TypeScript | 模块化强，自带 DI/校验/Swagger |
| 任务队列 | BullMQ + Redis | 长任务异步执行 |
| 实时通信 | Socket.IO | 任务进度实时推送 |
| 数据库 | PostgreSQL | 结构化存储，MVP 阶段单库即可 |
| 缓存 | Redis | 队列 + 缓存复用同一实例 |
| 向量检索 | pgvector 扩展 | 复用 PostgreSQL，避免额外运维向量数据库 |
| 文件存储 | 本地磁盘 + 火山引擎 TOS（生产） | MVP 用本地存储，部署时切 TOS |
| 模型服务 | 火山引擎 OpenAPI | 文生图、文生视频、图生视频、TTS |
| Monorepo | pnpm workspaces + Turborepo | 前后端共享类型定义 |
| CI/CD | GitHub Actions | 自动 lint + test + build + deploy |
| 部署 | Docker Compose + Nginx | 一键启动全部服务 |
| 测试 | Vitest（前端）+ Jest（后端）+ Playwright（E2E） | 分层测试 |

**选型原则**：MVP 阶段最小化基础设施依赖（PostgreSQL + Redis 两个有状态服务），避免过度工程化。

## 四、数据模型核心实体

在开始编码前明确核心数据结构，避免后期返工：

```
Material（素材）
├── id, merchant_id, type(image/video), url, filename
├── metadata: { category, tags[], source_declaration }
├── slices[]: { id, start_time, end_time, description, embedding }
└── created_at, updated_at

ReferenceVideo（参考视频/爆款库）
├── id, source_platform, source_url, category
├── analysis: { hook, selling_points, style, storyboard[] }
└── source_declaration（素材来源声明，课题要求）

Template（灵感模板）
├── id, name, strategy, factors{}
├── constraints[], applicable_categories[]
└── derived_from: reference_video_ids[]

Script（剧本）
├── id, merchant_id, product_info, template_id?
├── narrative_framework, visual_style
├── scenes[]: { order, description, camera_motion, duration,
│              dialogue, bgm_style, subtitle, constraints }
└── status: draft/confirmed

GenerationTask（生成任务）
├── id, script_id, status(queued/processing/done/failed)
├── progress: { current_step, total_steps, message }
├── result: { video_url, resolution, aspect_ratio, duration }
├── retry_count, error_log
└── created_at, completed_at

Video（成片）
├── id, task_id, merchant_id, script_id
├── url, thumbnail, duration, resolution, aspect_ratio
└── export_formats[]
```

## 九、关键工程难点与应对

| 难点 | 应对方案 | 降级策略 |
|------|----------|----------|
| **长任务管理** | BullMQ 队列 + Socket.IO 实时推送进度 + 数据库持久化任务状态 | 降级为轮询（3s 间隔） |
| **火山引擎 API 限流** | 请求队列 + 令牌桶限流 + 指数退避重试（最多 3 次） | 返回"排队中"状态，用户稍后查看 |
| **视频合成** | FFmpeg 服务端合成，分镜独立渲染后拼接 | MVP 阶段直接用火山引擎生成整段视频，不做本地合成 |
| **分镜编辑器交互** | 基于 dnd-kit 拖拽 + 自定义时间轴组件 | MVP 阶段用列表形式展示分镜，支持增删改 |
| **Prompt 稳定性** | JSON Schema 约束输出格式 + 结构化模板 + 失败自动重试换 Prompt | 预设 3 套 Prompt 模板轮换 |
| **前后端并行开发** | OpenAPI spec 先行 + Mock Server + 共享 TypeScript 类型 | — |
| **开发阶段无 API 额度** | 本地 Mock 模式：用预录制的 API 响应替代真实调用 | 录制一次真实响应，后续开发复用 |

## 十、Mock 与降级策略

为保证开发效率和演示稳定性：

1. **开发阶段 Mock**：
   - 火山引擎 API：录制真实响应存为 fixture，开发时直接返回
   - 视频生成：返回预制的示例视频 URL + 模拟 5s 延迟
   - 多模态理解：返回预设的标签和描述

2. **演示阶段降级**：
   - 如 API 不可用，展示预生成的视频作为 fallback
   - 数据看板使用 Mock 数据（课题允许）

3. **素材来源声明**：
   - 上传时必填"来源"字段（自有/公开可商用/参考）
   - 参考视频只保存结构化分析结果，不存储原视频
